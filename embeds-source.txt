SECTION 1: WHAT ARE EMBEDS?
================================================================================

In P-Stream's provider architecture, "embeds" (also called "embed scrapers") are
individual video extraction services that parse embed links returned by "sources"
(also called "sourcerers").

Think of it this way:
  Source (Sourcerer) → discovers embed URLs → Embed Scraper → extracts playable stream

A "source" is responsible for finding a movie/show on a streaming site and 
returning one or more embed URLs. An "embed" is responsible for taking that embed
URL and extracting the actual video stream (HLS/MP4) that the player can play.

Example flow:
  1. Source "RGShows" scrapes rgshows.me for movie "Inception"
  2. RGShows returns embed URLs like: https://vidcloud.co/embed/abc123
  3. Embed scraper "VidCloud" takes that URL and extracts the .m3u8 stream
  4. Player loads the .m3u8 and plays the video


SECTION 2: HOW EMBEDS WORK IN P-STREAM
================================================================================

The P-Stream library (@p-stream/providers) uses a builder pattern:

  const providers = buildProviders()
    .setTarget(targets.BROWSER)
    .setFetcher(standardFetcher)
    .setProxiedFetcher(proxiedFetcher)
    .addBuiltinProviders()     // adds ALL built-in sources + embeds
    .build();

When you call providers.runAll(media), the engine:

  1. INIT: Lists all registered sources that support the media type
  2. For each source (in order):
     a. START: Runs the source scraper → gets SourcererOutput
     b. DISCOVER_EMBEDS: If source returns embed URLs, lists them
     c. For each embed:
        - START: Runs the embed scraper on the URL
        - UPDATE: Reports success/failure with the extracted stream
  3. COMPLETED: Returns the first successful stream

Key types:
  - Sourcerer: { scrapeMovie, scrapeShow } → returns { embeds: [{embedId, url}] }
  - Embed: { scrape(url) } → returns { stream: [Stream] }
  - Stream: { type: 'hls' | 'file', qualities, captions, headers }


SECTION 3: SOURCE-ONLY vs EMBED-BASED PROVIDERS
================================================================================

Some sources return streams directly (no embeds needed):
  - FebBox: Custom source that calls API directly → returns MP4/HLS streams
  - Zeticuz: Custom source that parses HTML → returns MP4 download links

Most sources use embeds:
  - VidSrc, RGShows, etc. → scrape site → return embed URLs
  - Embed scrapers (VidCloud, Filemoon, etc.) extract streams from those URLs

The engine handles both cases automatically. If a source returns:
  - { embeds: [...] } → engine runs embed scrapers
  - { stream: [...] } → engine uses the stream directly


SECTION 4: HOW EMBEDS ARE USED IN THIS PROJECT (NEXUS)
================================================================================

NEXUS uses @p-stream/providers as its core provider library. The integration is:

  src/backend/providers/providers.ts
    → getProviders() builds the provider instance with all builtin providers

  src/hooks/useProviderScrape.tsx
    → useScrape() orchestrates the scraping process
    → Handles events: init, start, update, discoverEmbeds
    → Custom sources (FebBox) are added manually alongside library sources

  src/components/player/hooks/useSourceSelection.ts
    → Handles source and embed selection from the player settings menu
    → Allows manual source/embed switching
    → Calls providers.runSourceScraper() or providers.runEmbedScraper()

  src/components/player/atoms/settings/SourceSelectingView.tsx
    → UI for the source list in player settings
    → Shows all available sources with selection
    → Navigates to EmbedSelectionView when a source is chosen

The scraping flow in NEXUS:
  1. User plays media → useProviderScrape starts scraping
  2. FebBox is tried first (custom source, no embeds)
  3. If FebBox fails, regular providers run via runAll()
  4. Engine discovers embeds for each source
  5. First successful stream is loaded into the player


SECTION 5: EMBED-RELATED CONFIGURATION
================================================================================

Provider registration (src/backend/providers/providers.ts):
  - addBuiltinProviders() adds ALL built-in sources and embeds
  - Embeds are automatically paired with sources that return their IDs

Embed ordering (src/stores/preferences/index.tsx):
  - embedOrder: string[]       → preferred order of embed scrapers
  - enableEmbedOrder: boolean  → whether custom ordering is active
  - disabledEmbeds: string[]   → permanently disabled embed scrapers

Source vs Embed:
  - getCachedMetadata() returns metadata for both sources AND embeds
  - metadata.type === "source" for sources, "embed" for embeds
  - Each has: id, name, rank, flags, mediaTypes


SECTION 6: HOW TO ENABLE/REPAIR EMBED INTEGRATION
================================================================================

Step 1: Ensure all builtin providers are registered
  → providers.ts: addBuiltinProviders() must be called
  → This registers ALL sources AND all embeds from the library

Step 2: Check fetcher configuration
  → Fetchers handle HTTP requests through CORS proxies
  → Both standard and proxied fetchers must be configured
  → Proxy workers must be operational for CORS-blocked sources

Step 3: Verify embed discovery events are handled
  → useProviderScrape.tsx: discoverEmbedsEvent must update state
  → This allows the spinner to show which embeds are being tried

Step 4: Check for disabled sources/embeds
  → preferences store: disabledSources / disabledEmbeds
  → These arrays can prevent certain sources/embeds from running

Step 5: Verify extension integration
  → When browser extension is active, streams need preparation
  → prepareStream() sets up domain rules for HLS chunk loading


SECTION 7: MISSING LOGIC / IMPROVEMENTS
================================================================================

1. FAILED EMBED TRACKING PER MEDIA
   The project already tracks failedSourcesPerMedia and failedEmbedsPerMedia
   in the player store. When an embed fails for a specific movie/show, it's
   excluded from retry during that session.

2. EMBED RETRY WITH DIFFERENT SOURCE
   When "Find Next Source" is pressed, the engine skips the current source and
   tries the next one. Its embeds are then discovered fresh.

3. PROXY RELIABILITY
   Many embed scrapers need a working CORS proxy. The project uses load-balanced
   proxy URLs. If proxies are down, most embed-based sources will fail silently.

4. TOKEN-BASED EMBEDS
   Some newer embeds (e.g., Turnstile-protected) require browser-level challenge
   solving. The @p-stream/providers library includes a Turnstile global type for
   this purpose. Ensure the Turnstile script is loaded if needed.

5. CUSTOM SOURCE EMBEDS
   FebBox and Zeticuz bypass the embed system entirely. If you add a new custom
   source, decide whether it uses direct streams or embeds, and integrate
   accordingly.
